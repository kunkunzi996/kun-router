# 11 Computer-Use E2E Gate：高成本桌面验证门

## 目标

本 Gate 只负责 `12-verification-git-report.md` 验证阶梯中的 **V3 Computer-Use 桌面验证**。

V3 通过鼠标、键盘、截图和桌面操作模拟真实用户，是最后一级验证手段。它不能替代 V1 终端自动化验证或 V2 浏览器自动化验证。

本 Gate 默认关闭，高成本、按需触发，不因工具可用就自动运行。

## 启用前提

启用 V3 前必须同时满足：

1. 已执行 V1；适用时也已优先尝试 V2，或明确记录它们为什么无法覆盖关键路径。
2. 用户已明确授权本轮使用 Computer-Use，或项目验收契约已预先授权这一条具体路径。
3. 命中下面至少一个技术条件。

## 技术触发条件

- 原生桌面应用或需要跨多个 Windows 应用操作。
- 系统文件选择器、系统权限弹窗、原生菜单等 V2 无法控制的界面。
- 核心验收依赖拖拽、全局快捷键、跨窗口交互，且浏览器自动化无法可靠覆盖。
- 用户明确要求并授权“控制电脑，像真实用户一样操作一遍”。
- 部署后的关键路径无法通过浏览器自动化验证，并已记录具体原因。

## 默认不触发

以下情况本身不足以触发 V3：

- 普通网页的多页面、登录、表单、CRUD、上传下载、移动端适配或数据持久化；能用 V1/V2 覆盖就停在 V1/V2。
- 文案、样式、README、单文件小修、后端纯逻辑和已有明确自动化覆盖的 Bug。
- 功能仍在逐步施工中，尚未形成一条完整验收路径。
- 用户只要求规划、文档、规格或代码审查。

## 执行前检查

执行前先明确：

1. V1 / V2 已有证据，以及为什么仍需要 V3。
2. 本轮授权来源和准确的用户路径。
3. 当前环境是否支持桌面验证。
4. 是否需要测试账号、测试数据或环境变量。
5. 是否会触发真实线上行为。
6. 是否涉及付费、删除、群发、外部通知等高风险动作。

涉及真实付费、删除生产数据、群发通知、修改线上配置等动作，必须暂停并让用户确认。

## 成本控制

- 不在每个小步修改后重复控制电脑；一个完整功能完成后最多执行一次完整 V3 路径。
- V3 失败并修复后，只重跑失败路径和必要回归，不无条件重放全部步骤。
- 能降回 V1 / V2 获得同等证据时立即降级，不继续占用桌面。

## 规则

如果 E2E 验证失败，不能直接 Git 保存为完成状态。

如果当前环境无法进行 E2E 操作，必须明确说明：

- 哪一步没有验证。
- 为什么无法验证。
- 用户应该如何手动验证。
