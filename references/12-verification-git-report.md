# 12 Verification / Git / Report：验收、保存、汇报

## 分级验收

不要只根据代码阅读就说“完成”。选择**成本最低且足以证明本轮行为正确**的验证等级，不因为某个工具可用就自动升级。

### V1：终端自动化验证（默认先跑）

- 单元测试、集成测试、lint、类型检查、build、接口请求、数据库检查或文件结果检查。
- 能用 V1 充分证明后端、纯逻辑或非交互改动时，到此为止。

### V2：浏览器自动化验证（网页交互按需）

- 使用 Playwright 或等价浏览器自动化完成页面打开、点击、输入、跳转、截图和刷新检查。
- Web 页面需要交互证据且 V1 不够时使用；优先无头或独立测试浏览器，不接管整个 Windows 桌面。

### V3：Computer-Use 桌面验证（最后手段）

- 只在 V1 / V2 无法覆盖原生桌面、跨应用、系统弹窗或其它关键交互时使用。
- 必须满足 `11-computer-use-e2e-gate.md` 的启用前提，并取得本轮明确授权。
- 普通表单、CRUD、多页面或数据持久化本身不构成 V3 理由。

### 选择规则

1. 总是从 V1 开始；需要网页交互证据才升级 V2。
2. V1 / V2 已足够时禁止升级 V3。
3. 一个完整功能完成后再运行一次所需的最高等级，不在每个施工小步后重复真实操作。
4. 当前所需等级无法执行时，标记「未实际验证」并给用户手动路径，不得把降级说明写成“已通过”。

## 业务验收清单

### 前端 / UI

- 页面是否能打开。
- 样式是否正常。
- 按钮是否可点击。
- 移动端是否有明显问题。

### 数据功能

- 能否新增。
- 能否读取。
- 能否修改。
- 能否删除。
- 刷新后数据是否还在。

### Bug 修复

- 原 bug 路径是否修好。
- 附近正常流程是否仍然可用。

### 部署

- 本地能否启动。
- build 是否通过。
- 环境变量是否完整。
- 数据是否持久化。
- 公网地址是否可访问。
- 线上冒烟（部署成功 ≠ 完成）：打开线上真实地址，核心路径操作一遍、新功能生效、旧功能回归，四查全过才算部署完成（细则见 19-pipeline-loop.md）。

### 架构调整

- 旧入口是否还能用。
- 路由是否正常。
- API 是否兼容。
- 导入路径是否正确。
- 旧功能是否受影响。

## 无法验收时必须说明

- 哪些验收没跑。
- 为什么没跑。
- 用户应该如何手动检查。

## 开工前 Git 基线

如果项目即将进入首次代码施工，必须先检查：

```bash
git status
```

如果不是 Git 仓库，建议先建立初始存档：

```bash
git init
git add .
git commit -m "init project baseline"
```

对新手解释：

> Git commit 就是代码存档。正式施工前先存一个干净档，后面 AI 改坏了才能回退。

## 用户手动验收指引规则

完成报告里的「你可以这样验收」是必填项，按本轮类型给对应内容：

- 前端 / 页面 / 表单 / 数据功能：给点击路径（打开哪个页面、点哪个按钮、输入什么、保存后看哪里、刷新后数据是否还在）。
- Bug 修复：给原 bug 复现路径 + 修复后验证路径（同样的步骤走一遍，确认问题不再出现）。
- 后端 / API：给接口调用方式、请求示例、预期响应。
- 部署：给公网访问地址、要点开的核心页面、失败时怎么回滚或排查。

例外：

- 轻量小改（UI 小调、文案、单行样式）只给一两步轻量验收，看一眼即可，不用写满全套。
- 当前环境跑不起来时，明确标记「未实际验证」，再给用户照着点的手动步骤，不要假装跑过。

## Git 保存规则（挡位感知）

先读 `PROJECT_STATE.md` 的「流水线挡位」；没写挡位 = 手动挡。

### 手动挡（默认）

完成修改或验收后，默认不要直接执行 `git add` / `git commit` / `git push`。

默认流程：

1. 先输出修改文件清单、验收结果、当前 `git status`、建议 commit message、是否建议保存版本。
2. 等用户明确确认保存后，再执行 Git 保存命令。

只有在以下情况才允许实际执行 `git add` / `git commit` / `git push`：

- 用户明确说“保存一下 / 提交 Git / 帮我 commit / 同步到 GitHub”。
- 或用户在本轮开始前明确授权“完成并验收后自动提交”。
- 或当前任务本身就是“保存 / Git”。

用户确认后执行：

```bash
git status
git add .
git commit -m "简洁说明本次改动"
git push
```

如果没有远程仓库，只 commit，不假装 push。

如果当前目录不是 Git 仓库，直接说明。

禁止未经用户确认自动 commit / push。

### 自动挡（PROJECT_STATE 已声明）

先检查 PROJECT_STATE 的项目流水线契约；缺失或工具权限不可用时降级为手动挡。准入通过后，本轮初步验收全绿便进入 `19-pipeline-loop.md`；19 的 Step 1 按本节授权执行一次 `git add` / `git commit`，再同步基准分支并重新跑完整验收；复验全绿后继续 push → PR → CI → 合并 → 同步主工作区 → 部署 → 线上冒烟，不再逐步等确认。

自动挡也必须停的情况：流水线契约缺失；验收未全绿或存在「未实际验证」项；强制确认哨兵触发；工具或权限失效；CI 红灯 / 超时；主工作区同步失败；线上冒烟失败。

## 与项目洁癖 Gate 的衔接

`12-verification-git-report.md` 只负责验收、Git 和完成报告。

如果本轮满足以下任一条件，应继续启用 `13-project-cleanup-gate.md`：

- 功能完成或阶段完成。
- 部署完成。
- 架构、API、数据模型、目录结构、运行方式发生变化。
- 出现新的 AI 施工禁区或踩坑。
- 准备新开对话，需要下一轮 Codex 接手。
- 用户明确说“收尾 / 同步文档 / 项目状态 / 项目洁癖”。

小 UI、小文案、小样式修改默认不强制进入项目洁癖，除非影响项目说明或后续施工。

运行垃圾（已合并分支、worktree、临时数据）的清理不归 13 号门，走 19-pipeline-loop.md 的终态清理。
