# 03 Pre-Coding Gate：写代码前门禁

## 一句话定位

Pre-Coding Gate 用于正式写代码前，检查项目是否具备开工条件。

它不是传统 PRD，不负责长篇规划。

它只负责判断：**现在能不能进入 Codex 施工；如果不能，缺什么，应该路由到哪个 reference。**

## 核心原则

> 开工门禁只负责拦截和路由，不替代 Product Brief、AI-SDD、Git、设计门、架构门和项目洁癖。

## 使用时机

当用户出现以下表达时启用：

- 帮我开工这个项目。
- 可以开始写代码了。
- 让 Codex 开发。
- 准备进入施工。
- 这个设计稿能不能开始做了。
- 我已经有想法 / 页面 / 原型了，下一步怎么做。
- 新项目第一次进入开发。
- 已有项目新增大功能，且可能影响目录、数据、接口或架构。
- 自动挡项目准备进行任何实际修改；若只是低风险 UI / 文案 / 单行样式 / 单文件小修，则只运行「自动挡双项轻量门禁」。

## 不使用时机

以下情况默认跳过：

- 小 bug 修复（手动挡）。
- 小样式调整（手动挡）。
- 文案修改（手动挡）。
- 已有明确 Task Spec 的小任务（手动挡）。
- 单文件小修（手动挡）。
- 用户当前只想讨论想法，不准备进入施工。

## 自动挡双项轻量门禁

当任务满足 SKILL.md 的轻量路由条件，但项目挡位是自动挡时，不运行完整 7 项，只检查：

1. **分支工作区检查**：本轮 feature 分支 + worktree 是否已创建并核对绝对路径；
2. **流水线准入检查**：自动挡与项目流水线契约是否有效。

这不是把小改升级成完整开工门禁：目录、MVP、致命卡点、Spec、Git 五项标记为「轻量小改，不适用」，仍使用 3 块轻量路由报告和最小验收，不另外输出完整 Pre-Coding Gate 报告。双项检查结果压缩写进轻量报告的「本轮启用」；任一不通过则停止施工，通过后进入安全施工，完成后按 19 收口。

## 完整门禁必查 7 项

### 1. 项目目录是否清楚

至少确认：

```text
project-root/
├── PROJECT_STATE.md
├── docs/
│   ├── PRODUCT_BRIEF.md
│   ├── AI_SDD.md
│   ├── ARCHITECTURE_SNAPSHOT.md
│   └── TASKS/
├── src/ 或 app/
└── assets/ 可选
```

要求：

- 文档放 `docs/`。
- 代码放 `src/` 或 `app/`。
- 设计图、截图、素材放 `assets/`，可选。
- 不允许产品文档、设计图、代码全糊在一个目录里。

### 2. MVP 是否锁定

必须回答：

- 第一版只验证什么？
- 第一版必须做什么？
- 第一版明确不做什么？
- 如果这个功能做不通，项目是否还成立？

### 3. 致命卡点是否优先验证

必须找出：

- 本项目最可能失败在哪里？
- 哪个能力一旦做不通，整个项目就不成立？
- MVP 是否优先验证这个卡点？

原则：

> MVP 不是先做一个丑版本，而是先验证最危险的核心闭环。

### 4. 当前版本是否有 Spec

进入施工前，至少应该能找到或准备生成：

- `docs/PRODUCT_BRIEF.md`
- `docs/AI_SDD.md`
- `docs/TASKS/TASK_001_xxx.md`

如果没有，不允许直接写代码，应先回到：

- `04-product-brief-mvp.md`
- `05-ai-sdd-template.md`
- `06-task-spec-template.md`

### 5. Git 是否初始化

如果已经有代码目录，必须检查：

```bash
git status
```

如果不是 Git 仓库，提醒用户先初始化：

```bash
git init
git add .
git commit -m "init project baseline"
```

对新手解释：

> Git commit 就是代码存档。正式施工前先存一个干净档，后面 AI 改坏了才能回退。

### 6. 分支工作区检查

正式施工前必须确认这次改动在哪个分支、哪个工作区做，没确认前不许写代码。

判断规则：

- 新功能 / 大改 / 影响多个文件或页面，以及所有自动挡任务（包括小修）→ 默认「feature 分支 + 独立工作区（git worktree）」。
- 创建前先识别远程默认分支与远程名，不硬编码 main；再记录主工作区、feature worktree 的绝对路径：

```bash
git fetch <远程名>
git worktree add ../<项目名>-<功能名> -b feature/<功能名> <远程名>/<基准分支>
cd ../<项目名>-<功能名>
```

无远程的纯本地项目去掉 `git fetch`，并以本地 `<基准分支>` 作为 `git worktree add` 的起点；纯本地项目只能走手动挡收口。

- 手动挡的小修 / 小 bug / 单文件小改 / 改文案或样式 → 可以直接在当前分支当前目录改，不用新建。

强制三选一（用户不选不施工）：

1. 新建 feature 分支 + worktree 独立工作区（新功能 / 大改 / 自动挡默认）
2. 就在当前分支继续（只适合手动挡小修小补）
3. 我已经在正确的分支 / 工作区上了

对用户解释：

> 分支是草稿本里单开的一页；worktree 是把这页复印到另一张桌子上写——基准分支所在的主桌面始终干净可用，几个任务各占一张桌子互不打架，写完把结论誊回正本，桌子撤掉。

施工完成后，这条分支必须走 `19-pipeline-loop.md` 收口（初验 → commit → 同步基准分支并复验 → push → PR → CI → 合并 → 清理），不许留「有去无回」的分支。

### 7. 流水线准入检查

检查 `PROJECT_STATE.md` 是否已有「流水线挡位」字段：

- 已有手动挡 → 按记录执行，不再问。
- 已有自动挡 → 继续检查「项目流水线契约」和工具权限；完整才生效，缺失则列缺口并降级为手动挡。
- 没有 → 问一次二选一，答案写入 PROJECT_STATE：

1. 自动挡（推荐给已入门用户）——所有改动走 feature+worktree；补全流水线契约后，初验全绿自动 commit，再同步基准分支并复验；复验全绿后自动 push → PR → CI → 合并 → 部署 → 线上冒烟，汇报后等确认再终态清理。
2. 手动挡（新手 / 高危项目）——每个 Git / 上线动作都等确认。

自动挡必须补齐：基准分支、远程名、主工作区与 feature worktree 绝对路径、PR 工具、合并策略、CI 检查与超时、部署是否适用；适用时还要有目标环境、部署命令、线上地址、冒烟步骤、回滚命令与回滚模式；最后记录授权环境与确认日期。契约关键字段变化时准入状态重置为「待检查」。挡位定义与底线见 SKILL.md「流水线挡位」，执行细则见 19。

## 建议补充：AI 会话角色是否分工

> 这是**建议项**，不是完整门禁必查 7 项之一，不强制卡住施工（通过标准里也不含此项）。复杂项目建议做，单文件小修可跳过。

正式开发前，建议至少拆分两个 AI 会话：

1. 产品 / Router 会话  
   负责 MVP、里程碑、Spec、验收标准。

2. 研发 / Codex 会话  
   只负责按 Spec 小步施工，不重新发散需求。

复杂项目可增加：

3. UI 会话  
   负责视觉风格、页面布局、设计基调。

4. QA 验收会话  
   负责对照 Spec 检查功能是否真的完成。

## 设计沉迷拦截

如果用户持续打磨高保真 UI、logo、颜色、动效，但核心功能尚未验证，必须提醒：

> 当前可能陷入设计细节。建议先回到 MVP，验证核心功能和致命卡点，视觉细节放到 v0.2 / v0.3。

## 通过标准

### 完整门禁

只有满足以下条件，才允许进入施工：

- 目录结构清楚。
- MVP 已锁定。
- 致命卡点已识别。
- 当前 Spec 已存在或正在生成。
- Git 状态清楚。
- 基准分支、分支与工作区已确认（feature+worktree / 当前分支 / 不适用）。
- 流水线挡位已声明；自动挡已通过项目流水线契约准入，或已明确降级为手动挡并记录缺口。
- 本轮只做 / 不做已写明。

### 自动挡双项轻量门禁

只要求「分支工作区检查」和「流水线准入检查」通过。目录、MVP、致命卡点、Spec、Git 五项明确标记为「轻量小改，不适用」，不得因为没跑 MVP / Spec / 架构检查而判定失败。

## 不通过时的路由

不要直接施工。

按缺口路由：

| 缺口 | 路由到 |
|---|---|
| 想法是否值得做不清楚 | `01-idea-pressure-test.md` |
| 没有开工规格 | `02-spec-start-qiaomu-inspired.md` |
| 缺 MVP / 第一版边界 | `04-product-brief-mvp.md` |
| 缺 AI-SDD | `05-ai-sdd-template.md` |
| 缺本轮任务规格 | `06-task-spec-template.md` |
| 页面视觉没基调 | `07-design-gate.md` |
| 涉及架构 / API / 数据 / 部署 | `08-architecture-gate.md` |
| 缺测试或验收标准 | `09-test-first-gate.md` |
| 准备让 Codex 改代码 | `10-codex-safe-construction.md` |
| 缺 Git / 验收 / 保存方式 | `12-verification-git-report.md` |


自动挡小改不单独使用这份完整模板，只在 3 块轻量报告「本轮启用」里写：`03 自动挡双项轻量门禁：分支工作区检查通过；流水线准入检查通过`。

## 禁止行为

- 不允许没有 MVP 就直接开发大功能。
- 不允许没有 Spec 就让 Codex 大改。
- 不允许文档、代码、素材混在一个目录。
- 不允许先沉迷高保真 UI，而不验证核心功能。
- 不允许为了走流程而让小 bug、小文案、小样式改动变重。
