# 13 Project Cleanup Gate：项目三层知识编辑门

## 一句话定位

项目洁癖门用于已验收功能、阶段或部署的收尾。

它不是“把文档写得更多”，而是核对项目知识是否仍然真实、可用、可交接。

它只处理当前代码项目，不处理 Obsidian、个人知识库、人生复盘或无关素材。

## 目录

1. 使用时机与边界
2. 三层知识模型
3. 触发时要读什么
4. 权限与删除红线
5. 八步执行流程
6. 自检与报告

## 使用时机

满足任一条件时启用：

- 一个功能完成并已经验收。
- 一个 Bug 已按复现路径验证修复。
- 一个阶段完成，准备进入下一阶段。
- 部署完成，或运行、部署方式发生变化。
- 架构、API、数据模型、目录、环境变量或运行命令发生变化。
- 出现需要长期遵守的 AI 施工规则或高复用踩坑。
- 准备换窗口，需要下一轮 Agent 快速接手。
- 用户明确说“收尾”“项目洁癖”“同步文档”“整理项目”且有项目开发语境。

以下情况默认不启用：

- 纯文案或纯样式小改。
- 临时实验尚未进入主线。
- 尚未验收的任务。
- 只讨论方案、没有改动项目。
- 用户明确说不要更新项目文档。

小改若影响运行方式、稳定规则、下一轮入口或项目说明，仍可启用最小范围的洁癖。

## 三层知识模型

### 1. 项目级 Agent 记忆

范围是能可靠映射到当前项目的 Agent 记忆。

它保存近期教训、非显而易见的临时上下文和项目指针。

它不是长期架构说明，也不是无边界的历史仓库。

### 2. 项目 AI 规则

范围是项目内 `AGENTS.md`、`CLAUDE.md`、局部规则文件和明确归属的 Agent 规则。

它保存红线、常用命令、目录责任、权限边界和稳定施工约定。

它不是变更日志，不记录某一次开发的流水账。

### 3. 项目文档与状态

范围是 `README.md`、`docs/`、`PROJECT_STATE.md`、`CONTEXT.md`、`ACCEPTANCE.md`、`DECISIONS.md` 和 `HANDOFF.md`。

它面向人类、下游开发者和下一轮接手 Agent，保存当前架构、使用方式、验收、决策和交接。

三层主责不同，但迁移期间允许保留必要的摘要或指针。

同一事实必须有一个权威落点，不能在三层中各写一份互相矛盾的版本。

用户级全局规则不属于当前项目三层；个人知识库和 Obsidian 永远不纳入本门。

## 触发时要读什么

先读 `12-verification-git-report.md`，确认本轮已验收、修改范围和 Git 状态。

主流程只读本文件。

出现下列情况时，再按需读取配套 reference：

| 情况 | 读取文件 | 原因 |
|---|---|---|
| 发现 Agent 记忆、项目规则或需要判断平台路径 | `13-project-cleanup-platforms.md` | 区分项目级、全局和机器生成状态 |
| 发生功能、API、数据、部署、规则审计或反向清理 | `13-project-cleanup-matrix.md` | 使用同步矩阵和审计证据格式 |
| 只做一个按钮文案小改 | 不读取配套 reference | 保持轻量路由 |

不要因为文件存在就全文读取全部知识库。

未发现可明确归属的项目级 Agent 记忆时，将记忆层标记为“不适用”，不要扫描全局平台目录。

## 权限与删除红线

可以直接做的事只限于当前项目中有证据支持、可逆且不丢信息的修改：

- 修正被代码、配置、测试或当前文档证明已经过期的事实。
- 补充已经真实存在的命令、路径、字段或验收方式。
- 合并完全重复且不丢信息的文字。
- 把明确的相对时间改为绝对日期。
- 更新项目状态和下一轮入口。

以下动作必须列入“待你拍板”，不能自动执行：

- 删除文件、目录、文档段落、历史记录或记忆内容。
- 重命名、移动、合并文件，或创建、修改软链接。
- 把原记忆缩成一行指针。
- 修改用户级全局 `AGENTS.md`、`CLAUDE.md` 或其他全局规则。
- 修改其他项目或下游项目。
- 安装依赖、改环境、改 Git 配置或修改部署方式。
- 无法判断哪份规则权威，或规则漂移需要改变既有约定。

用户确认删除后，也只能一次处理一个明确路径，禁止批量删除。

## 八步执行流程

### Step 0：确认已验收与权限范围

先确认：

- 本轮做了什么、修改了哪些文件。
- 验收是否通过，哪些检查尚未执行。
- 当前 Git 分支和工作区状态。
- 只处理当前项目，还是用户已明确授权多个项目。
- 是否允许读取可明确归属的项目级 Agent 记忆。

未验收的任务不做“已完成”同步。

需要删除、改全局规则、改其他项目或改部署时立即暂停。

### Step 1：平台探测和尺寸体检

先确认项目规则、记忆和文档属于谁，再记录尺寸。

至少记录：

- 项目 `AGENTS.md`、`CLAUDE.md` 和局部规则的行数、字节数。
- 可明确归属的 Claude `MEMORY.md` 的行数、字节数。
- 单条项目记忆的行数或大小。
- `docs/` 中明显过大的文件。

体检阈值只用于提醒，不替代事实判断：

| 对象 | 阈值类型 | 建议 |
|---|---|---|
| Claude `MEMORY.md` | 平台硬限制 | 不超过 200 行且不超过 25KB；约 70% 时预警 |
| 项目 `AGENTS.md` / `CLAUDE.md` | 项目软提醒 | 超过约 200 行或 15KB 时检查历史叙事和重复规则 |
| 单条项目记忆 | 项目软提醒 | 超过约 100 行时检查是否混入多件事 |
| docs 单文件 | 项目软提醒 | 超过约 1500 行时建议拆分，拆分动作待拍板 |

不要把“记忆比文档大”当成自动失败条件。

### Step 2：机械式文件盘点

先完整枚举，再分层读取。

枚举范围包括：

- 项目根目录 Markdown。
- `docs/` 下 Markdown。
- `PROJECT_STATE.md`、`HANDOFF.md`、`CONTEXT.md`、`ACCEPTANCE.md`、`DECISIONS.md`。
- 项目内 `CLAUDE.md`、`CLAUDE.local.md`、`AGENTS.md`、`AGENTS.override.md`。
- `.claude/rules/` 下的规则文件。
- 能明确映射到当前项目的 Agent 记忆。

默认排除 `.git/`、依赖目录、构建目录、虚拟环境和第三方生成目录。

每个文件都标记为以下之一：

1. 已枚举-待评估。
2. 已评估-不用改。
3. 要改。
4. 已改。
5. 阻塞/超大/无权限。

“已枚举”不等于“已阅读”，不能假装全部评估过。

### Step 3：建立事实证据

判断事实时按以下顺序取证：

```text
可运行代码 / 配置 / schema / API 实际响应 / 测试结果
→ 当前有效的决策与验收文档
→ Git 提交和变更记录
→ 项目规则
→ 会话陈述
→ Agent 记忆
```

高优先级证据与低优先级文字冲突时，以高优先级证据为准。

仍然无法判断时，标为“待你拍板”，不要擅自选择权威版本。

### Step 4：规范执行审计

检查规则是否仍可执行，而不是只看规则写得是否漂亮。

每条审计发现必须包含：

- 规则出处：文件与小节。
- 现实证据：命令结果、路径、代码位置或测试结果。
- 判断：实践违规、死引用、规则矛盾或规则漂移。
- 建议动作：修正文档、补充规则、待拍板或仅报告。
- 是否可以直接修，以及对外影响。

同一规则反复被违反时，只建议后续 hook 化，不自动改 Git 配置。

### Step 5：正向同步和反向清理判断

先问“本轮新事实应该同步到哪里”，再问“哪些旧内容已经过期、重复或错位”。

发生功能、API、数据、部署、规则审计或清理判断时，读取 `13-project-cleanup-matrix.md`。

跨项目只报告受影响项目和建议修改位置。

没有用户明确授权时，不能写下游项目。

### Step 6：按权威顺序修改

修改顺序固定为：

1. 项目事实文档：`docs/`、`README.md`。
2. 项目 AI 规则：项目内 `AGENTS.md`、`CLAUDE.md` 和局部规则。
3. 项目级 Agent 记忆：仅在平台允许、归属明确且无需删除时。
4. 状态收口：`PROJECT_STATE.md`、`HANDOFF.md`、`CHANGES.md`。

这样不会先在状态文件里声称“已同步”，却在后续修改中失败。

不要为了形式创建空壳 `PROJECT_STATE.md`、`CONTEXT.md`、`ACCEPTANCE.md` 或 `DECISIONS.md`。

### Step 7：记忆毕业机制

满足以下任一条件的项目级记忆，可以列为“毕业候选”：

- 同一主题连续出现三次。
- 内容解释系统的长期工作方式。
- 内容是已经落地的事件记录。
- 下一位接手者也必须知道。

毕业分两步：

1. 先把准确内容写入项目文档或项目规则，并验证新落点完整。
2. 原记忆的删除、缩短或改成指针一律列入待拍板。

用户未确认前，不得因为“毕业”丢失原内容。

### Step 8：自检和报告

完成前确认：

- 每个已枚举文件都有最终状态。
- 每项修改都有事实证据。
- 没有自动删除文件、目录、段落或记忆。
- 没有改全局规则、个人知识库或其他项目。
- 没有把机器生成的 Codex 记忆当作可手改项目文档。
- 报告只列实际变化、阻塞和需要用户决定的项目。
- 下一轮入口包含当前状态、继续文件、验收方式和风险。

```md

## 本轮范围
- 当前项目：
- 允许读取的项目记忆：是 / 否 / 不适用
- 已验收证据：

## 三层体检
- 项目级 Agent 记忆：
- 项目 AI 规则：
- 项目文档与状态：

## 文件盘点
- 已评估-不用改：
- 已改：
- 阻塞/无权限：

## 规范审计
- 规则出处：
- 现实证据：
- 判断与建议：

## 已同步
- 文档：
- 项目规则：

- 默认未处理 Obsidian、个人知识库和不明归属的记忆。
```

## 最重要原则

宁可少写，也要准确。

宁可报告“未同步原因”，也不要假装所有知识都已经更新。

当前项目内部闭环优先，删除与越界动作永远等待用户确认。
