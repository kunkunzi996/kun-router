# 13 Project Cleanup：同步矩阵与规范审计

## 一句话定位

本文件只在发生功能、API、数据、部署、规则审计或反向清理时读取。

它给出“新事实同步到哪里”“旧内容怎么处置”和“审计发现如何取证”的统一格式。

## 目录

1. 正向同步矩阵
2. 反向清理矩阵
3. 规范执行审计
4. 跨项目影响

## 正向同步矩阵

| 新事实 | 优先权威落点 | 可保留的摘要 | 不应自动做 |
|---|---|---|---|
| 新增或移除用户功能 | README、AI_SDD、PROJECT_STATE、CHANGES | AGENTS 中的入口提示 | 删除旧功能历史 |
| MVP 边界变化 | PRODUCT_BRIEF、AI_SDD、PROJECT_STATE、DECISIONS | Handoff 中的本轮提醒 | 推翻未确认决策 |
| 页面或交互变化 | DESIGN_SPEC、AI_SDD、E2E 路径 | README 功能概览 | 新建无用设计文档 |
| 架构或目录变化 | ARCHITECTURE_SNAPSHOT、项目规则 | PROJECT_STATE 的当前阶段 | 重命名目录 |
| API 或数据模型变化 | ARCHITECTURE_SNAPSHOT、AI_SDD、CONTEXT | README 的使用变化 | 修改下游项目 |
| 运行命令或环境变量变化 | README、项目规则、ACCEPTANCE | PROJECT_STATE 当前入口 | 修改全局环境 |
| 部署方式变化 | README、ACCEPTANCE、ARCHITECTURE_SNAPSHOT | HANDOFF 风险提醒 | 部署或改生产配置 |
| 关键 Bug 修复 | TROUBLESHOOTING、ACCEPTANCE、CHANGES | PROJECT_STATE 已完成 | 删除旧故障记录 |
| 新施工红线 | 项目 AGENTS/CLAUDE、PROJECT_STATE | Handoff 禁止事项 | 修改全局规则 |
| 稳定记忆毕业 | 对应项目文档或项目规则 | 原记忆指针 | 缩短或删除原记忆 |

找不到对应文件时，先报告“建议新增的权威落点”和原因。

不要为了填满表格而创建空壳文档。

## 反向清理矩阵

| 发现 | 现实证据 | 默认动作 | 是否需拍板 |
|---|---|---|---|
| 文档事实已过期 | 代码、配置、测试或实际响应 | 修正为当前事实并记录依据 | 否，前提是证据明确 |
| 相对时间失效 | 文件内容与当前日期 | 改为绝对日期 | 否 |
| 完全重复且不丢信息 | 两处文本逐项一致 | 合并到权威落点 | 否，保留信息 |
| 同一事实多处矛盾 | 高低优先级证据冲突 | 以高优先级证据修正，无法判断则报告 | 可能 |
| AGENTS/CLAUDE 夹带流水账 | 项目规则中有历史叙事 | 建议迁移目标，不自动删除原文 | 是 |
| 失效规则或死引用 | 文件路径、命令或链接不存在 | 修正规则或列出待定 | 证据明确时否 |
| 目录、文件或记忆应移除 | 发现冗余但仍可能有历史价值 | 只列候选 | 是 |
| 记忆内容已毕业 | 文档新落点已验证 | 保留原内容并列压缩候选 | 是 |

“清理”不等于“删除”。

在用户确认前，保留原文件、原段落和原记忆。

## 规范执行审计

每条发现都使用下面的最小证据格式：

```md
### 发现：<简短标题>
- 规则出处：<文件 + 小节>
- 现实证据：<命令结果、路径、代码位置或测试>
- 判断：实践违规 / 规则死引用 / 规则矛盾 / 规则漂移
- 建议动作：<修正、补充、仅报告或待拍板>
- 可直接修：是 / 否，原因：
- 外部影响：当前项目 / 下游项目 / 全局规则 / 无
```

分类说明：

| 分类 | 含义 | 默认处置 |
|---|---|---|
| 实践违规 | 已有规则与现实操作不一致 | 修正实践或报告证据 |
| 规则死引用 | 规则指向不存在的文件、命令或流程 | 修正规则，前提是事实明确 |
| 规则矛盾 | 两条有效规则相互冲突 | 列出两条规则与证据，待拍板 |
| 规则漂移 | 规则曾合理但当前事实已变 | 以权威证据更新，必要时待拍板 |

反复出现的同类违规可以建议后续用 hook 防护，但不自动安装 hook 或修改 Git 配置。

## 跨项目影响

发现下游项目或其它仓库受影响时，只报告：

- 受影响项目或接入方。
- 可能过期的文档、接口约定或验收项。
- 建议由谁确认、应修改哪里。

未获得用户明确多项目授权前，不读取或写入另一个项目。

## 最重要原则

矩阵用于防漏项，不用于扩大修改范围。

每项同步和清理都必须能追溯到现实证据。
