# 10 Codex Safe Construction：安全施工

用于所有会改代码、配置、文档、部署文件的任务。

## 开工前必须说明

- 当前目录。
- 当前分支。
- 本轮只做什么。
- 本轮不做什么。
- 预计修改哪些文件。

## 施工顺序

1. 先读项目结构和相关文件。
2. 再给最小施工计划。
3. 每次只做一个小任务。
4. 只修改直接相关文件。
5. 不乱格式化。
6. 不隐藏重构。
7. 不乱装依赖。
8. 不批量删除文件。
9. 不擅自改数据库结构。
10. 不擅自改部署方式。
11. 不擅自暴露公网。
12. 不擅自修改密钥、环境变量或凭证。

## 奥卡姆剃刀：如无必要，勿增实体

只实现本轮任务 / 已确认计划明确要求的能力，不新增计划外的用户能力、架构机制或顺手优化。

- 额外缓存、额外配置层、国际化、重试机制、"顺便重构"等计划外扩展，一律不写进代码。
- 为保证正确性、安全性、兼容性、可访问性、测试和既有项目规范所必需的最小实现，不属于计划外扩展。
- 必要实现如果会突破已确认范围或文件白名单，必须暂停让用户确认；不阻塞本轮的优化想法，写进完成报告的「建议」里，下一轮再做。
- 判据一句话：范围、验收和项目规则都不要求，且不是正确完成任务所必需的能力，就是"实体"，不要增。

## 施工聚焦：执行中不跑题

进入施工执行后，只输出必要的简短进度，不插播计划外的概念解释、可选建议或闲聊。

- 非阻塞建议先记下，收进完成报告或待拍板清单统一说。
- 阻塞问题、测试失败、高风险动作、用户新指令，以及会影响正确性或范围的新事实，必须立即说明，不得压到完成报告。
- 触发「高风险动作必须暂停确认」时照常暂停提问；平台要求的必要进度更新照常输出。
- 完成报告不受本条限制：仍按 `12-verification-git-report.md` 给大白话验收指引。

## 高风险动作必须暂停确认

- 安装新依赖。
- 改架构。
- 改数据库。
- 改 API 协议。
- 改目录结构。
- 替换核心依赖。
- 上传部署。
- 开公网隧道。
- 删除文件。
- 修改密钥或环境变量。

## 小步提交原则

一次只完成一个可验收的小目标。

不要把功能开发、UI 美化、重构、依赖升级、部署改动混在同一个任务里。
