# 09 Computer-Use E2E Gate：计算机操作端到端验证门

## 目标

通过真实用户视角验证功能是否真的可用，而不是只根据代码、单元测试、构建结果或静态检查判断完成。

当项目涉及复杂交互、多页面流程、桌面应用、数据持久化、部署访问等场景时，应使用计算机操作型端到端验证。

本 Gate 不是每次必跑，而是一个高成本、按需触发的验证门。

## 适用背景

参考类似 Warp 使用内部 UX 验证工具的思路：在可控环境中赋予 Agent 鼠标、键盘、浏览器或桌面操作权限，让 Agent 模拟真实用户完成操作链路。

它弥补的是单元测试、接口测试、构建检查无法覆盖的问题：

- 页面真的能不能点
- 表单真的能不能提交
- 弹窗、拖拽、键盘输入是否符合预期
- 多页面跳转是否顺畅
- 数据从输入到展示再到持久化是否完整
- 桌面应用或复杂交互是否符合真实用户体验

## 触发条件

以下情况建议开启 Computer-Use E2E Gate：

- 多页面用户流程
- 登录 / 注册 / 权限
- 表单提交
- 数据新增 / 编辑 / 删除
- 文件上传 / 下载
- 拖拽、弹窗、快捷键、复杂交互
- 桌面应用
- 移动端适配
- 部署后公网访问
- 数据持久化验证
- 单元测试无法覆盖完整用户路径
- 用户明确要求“像真实用户一样操作一遍”

## 不触发条件

以下情况默认跳过：

- 文案小改
- 样式微调
- README 修改
- 单文件小修复
- 后端纯逻辑修改
- 已有明确测试覆盖的小 bug
- 用户当前只要求规划、写文档、写规格，不进入真实操作验证

## 执行前检查

执行前先明确：

1. 当前环境是否支持真实操作验证
2. 是浏览器页面、桌面应用、小程序模拟器，还是云端沙箱
3. 是否需要测试账号、测试数据或环境变量
4. 是否会触发真实线上行为
5. 是否涉及付费、删除、群发、外部通知等高风险动作

涉及真实付费、删除生产数据、群发通知、修改线上配置等动作，必须暂停并让用户确认。

## 执行方式

1. 先定义用户操作路径。
2. 明确起始状态、测试数据和预期结果。
3. 在浏览器、桌面环境或云端沙箱中，以真实用户方式操作。
4. 尽量使用鼠标、键盘、点击、输入、滚动、刷新等真实行为。
5. 记录每一步是否通过。
6. 如果失败，记录失败步骤、现象和可能原因。
7. 修复后重新跑同一条用户路径。

## 常见验证案例

### Web 工具类产品

- 打开首页
- 新增一条记录
- 在列表中看到记录
- 编辑记录
- 刷新页面
- 确认数据仍然存在
- 删除记录
- 确认列表状态正确

### 仪表盘 / 排行榜

- 打开仪表盘
- 切换筛选条件
- 查看排序结果
- 点击详情
- 返回列表
- 确认筛选状态或结果符合预期

### 桌面应用 / 复杂交互

- 启动应用
- 使用鼠标点击关键入口
- 使用键盘输入或快捷键
- 完成一条核心任务链路
- 关闭并重新打开
- 确认状态和数据符合预期

## 通过标准

Computer-Use E2E Gate 通过需要满足：

- 核心用户路径完整跑通
- 页面或应用没有明显崩溃
- 用户输入能被正确处理
- 关键数据能正确展示或保存
- 刷新、重启或返回后状态符合预期
- 未破坏本轮任务之外的核心流程

## 失败处理

如果 E2E 验证失败，不能直接 Git 保存为完成状态。

必须记录：

- 失败发生在哪一步
- 用户看到了什么
- 预期应该是什么
- 初步判断可能原因
- 下一步修复范围

修复后重新执行同一条用户路径。

## 无法执行时必须说明

如果当前环境无法进行真实计算机操作验证，必须明确说明：

- 哪一步没有验证
- 为什么无法验证
- 用户应该如何手动验证
- 是否可以退化为普通手动验收清单

## 与其他 Gate 的关系

Computer-Use E2E Gate 不替代 Test First Gate。

推荐顺序：

```text
Test First Gate
→ Codex 安全施工
→ Computer-Use E2E Gate（按需）
→ Verification / Git / Report
```

Test First Gate 负责提前定义测试和验收标准。  
Computer-Use E2E Gate 负责像真实用户一样操作一遍。  
Verification / Git / Report 负责最终保存和汇报。
